# Migrating from 0.0.x to 1.0

This guide enumerates every breaking change between the 0.0.x
experimentation phase and the first stable release. Each entry shows
the before-and-after snippet and the reasoning.

> Most consumers on 0.0.x are internal; the breaking changes below
> are listed for completeness, not because they're widespread. If you
> hit something not on this list, please open an issue.

## TL;DR

```bash
# 1. Update every @konfig.ts/* dep to ^1.0.0 (lockstep version).
# 2. Run `konfig build prod` once; the CLI will surface most issues.
# 3. Skim the categorized changes below for the type errors you'll see.
```

## 1 · `SecretBackend` is now `SecretBackend<N, K, RequiresSource>`

**Before:**

```ts
import type { SecretBackend } from "@konfig.ts/k8s";

declare const sopsBackend: SecretBackend<"db", "url" | "password">;
Secret.bind({ secret: dbCreds, backend: sopsBackend });
// ☝️ TypeScript was happy. Render threw `BackendSourceMissing` at runtime.
```

**After:**

```ts
import type { SecretBackend } from "@konfig.ts/k8s";

declare const sopsBackend: SecretBackend<"db", "url" | "password", true>;
Secret.bind({ secret: dbCreds, backend: sopsBackend, source: sopsSource });
// ☝️ Backend types now carry `RequiresSource`; missing `source` fails to compile.
```

**Why:** `Sops.backend`, `SealedSecrets.backend`, and
`NativeSecret.backend` all need a plaintext source at render time.
Encoding `requiresSource` in the type makes the omission a compile
error instead of a render-time `BackendSourceMissing` throw.

**Migration:** if you wrote your own backend, decide whether you need a
source and pick `true` or `false`. Most users don't write their own and
will only see this in error messages.

## 2 · `coerce<T>(value)` → `unsafeCoerce<T>(value, reason: string)`

**Before:**

```ts
import { coerce } from "@konfig.ts/core";

const obj = coerce<Record<string, unknown>>(parsed);
```

**After:**

```ts
import { unsafeCoerce } from "@konfig.ts/core";

const obj = unsafeCoerce<Record<string, unknown>>(
  parsed,
  "parsed YAML is structurally Record<string, unknown>; refined by checks below",
);
```

**Why:** every unsafe cast deserves a one-line justification readable
in code review. `coerce<T>` is kept as a deprecated alias so existing
test fixtures keep compiling; production code should migrate.

**Migration:** grep `coerce<` and rewrite each call. For boundary
values (sops/kubeseal/process output), prefer `boundary(schema)` over
`unsafeCoerce`.

## 3 · `Application.define`'s `name` / `namespace` require literals

**Before:**

```ts
const myName: string = process.env.APP_NAME ?? "api";
Application.define({ name: myName, namespace: "app", ... });
// ☝️ Compiled. Silently widened Dep.Provide<"App", string>.
```

**After:**

```ts
const myName: "api" = "api";
Application.define({ name: myName, namespace: "app", ... });
```

**Why:** widening to `string` collapses every Application's
`Provide<"App", N>` slot into a single key. The
`LiteralName<T>`-branded error type makes the widening a compile error
at the call site.

**Migration:** if you've been passing a string variable, hoist it into
a literal type or use `Module.fixedNs`/`Module.dynamicNs` (both of
which apply `LiteralName` at every layer).

## 4 · `Manifest<A>` lost its `R` and `P` parameters

**Before (the 0.0.0 README):**

```ts
const m: Manifest<MyValue, { Secret: "creds" }, { Namespace: "app" }>;
```

**After:**

```ts
const m: Manifest<MyValue>;
// Per-kind dep tracking moved to Effect Layer<Out, _, In> at the
// Application boundary. See docs/architecture.md.
```

**Why:** the per-kind R/P parameters were a stand-in for what Effect
`Layer<Out, _, In>` does more cleanly. The Layer composition machinery
also gets us free Context, error channels, and an existing semantics
for `provideMerge`.

**Migration:** drop the R and P parameters from any explicit
`Manifest<A, R, P>` annotations. Dep-graph tracking now happens via
`Application.define`'s `Out`/`In` channels.

## 5 · `Yaml.serialize` and `diffFiles` use object inputs

**Before:**

```ts
Yaml.serialize(myResource);
diffFiles(leftMap, rightMap);
```

**After:**

```ts
Yaml.serialize({ value: myResource });
diffFiles({ left: leftMap, right: rightMap });
```

**Why:** house style enforces named-args via `app/no-multiple-function-params`.

**Migration:** mechanical; the codemod is a regex.

## 6 · `Helm.release` requires `digest`, verified after pull

**Before:** `digest` was optional, used only as a cache buster.

**After:** `digest` is required. It must be `sha256:<64 hex chars>`.
The verifier hashes the cached `.tgz` against it on every load.
Mismatch fails with `HelmDigestMismatch`.

**Migration:**

```bash
# Find the real digest for each pinned chart:
curl -sL https://charts.example.com/foo-1.0.0.tgz | shasum -a 256
```

Then paste into `digest: "sha256:..."`. For the OCI registry case
where you receive a base64 digest, the upcoming
[`konfig helm digest <repo>/<chart>@<ver>`](../packages/cli/README.md)
helper will compute the canonical hex form for you.

## 7 · Multiple CLI flag additions and atomic file writes

`konfig build`/`validate`/`diff` accept `--cluster`, `--k8s-version`,
`--flag k=v`. `konfig build` also emits a per-phase timing report on
stdout (or JSON via `--log=json`).

`konfig build` writes through `<outDir>.tmp` and renames atomically.
If you've been scripting around partial-write recovery, you can simplify.

## 8 · `coerce<T>` → `unsafeCoerce<T>` (lint enforcement)

The `app/no-multiple-function-params` rule now defaults to `scope:
"exports"` so internal helpers may take positional args. If you
overrode this in your own oxlint config, drop the override.

## 9 · `defineEnvironment` collisions caught at compile time

```ts
defineEnvironment({
  a: defineLiteral({ envName: "SHARED", value: "x" }),
  b: defineLiteral({ envName: "SHARED", value: "y" }),
});
// ❌ envName "SHARED" is claimed by multiple members
```

**Migration:** rename the colliding `envName` (the runtime check was
already there — the change is just that you'll see it sooner now).

## 10 · `kubernetes-types` pinned to an exact patch

**Before:** `"kubernetes-types": "^1.30.0"`.
**After:** `"kubernetes-types": "1.30.0"` (exact).

If you depend on `@konfig.ts/k8s` you'll resolve the pinned version
transitively. If you also installed `kubernetes-types` directly, pin
to the same version to avoid the rare nominal-type mismatches.

## Where to file issues

If a migration step is missing or doesn't work for your code, file an
issue at https://github.com/DwieDave/konfig.ts/issues with the
"migration" label.
